# Deploying BeastBots on Render

This guide covers how to deploy the BeastBots API and Web services on [Render](https://render.com) using the included `render.yaml` blueprint.

---

## Architecture on Render

BeastBots deploys as **two services** from the monorepo:

| Service | Dockerfile target | Default port | Purpose |
|---|---|---|---|
| `beastbots-api` | `api` | 4000 | Hono REST API, SQLite database, bot runtime |
| `beastbots-web` | `web` | 3000 | Next.js command-center UI |

Both targets are defined in the multi-stage `Dockerfile` at the repo root.

---

## Quick Start

1. **Fork / push** this repo to your GitHub account.
2. In the Render dashboard go to **Blueprints → New Blueprint Instance**.
3. Connect your repo and select the branch to deploy.
4. Render reads `render.yaml` and creates both services plus a persistent disk.
5. Set the environment variables listed below, then trigger a manual deploy.

> **Tip:** `JWT_SECRET` and `ENCRYPTION_KEY` are auto-generated by the blueprint. You still need to fill in the networking and optional integration variables described below.

## Environment Variables

### Required (API will not start without these)

| Variable | Where | Description |
|---|---|---|
| `JWT_SECRET` | API | Secret key for signing JWTs. Auto-generated by the blueprint; override with a strong random value (≥ 32 chars). |
| `ENCRYPTION_KEY` | API | Key used to encrypt stored credentials (AES). Auto-generated; override with a strong random value (≥ 32 chars). |

### Networking / Service URLs

Set these **after both services are created** so you know the Render-assigned URLs.

| Variable | Where | Example | Description |
|---|---|---|---|
| `FRONTEND_URL` | API | `https://beastbots-web.onrender.com` | Allowed CORS origin for the API. Set to the Web service URL. |
| `API_BASE_URL` | API | `https://beastbots-api.onrender.com` | The API's own public URL (used in emails, webhook callbacks, etc.). |
| `NEXT_PUBLIC_API_URL` | Web | `https://beastbots-api.onrender.com` | **Build-time variable.** The URL the browser uses to reach the API. Trigger a rebuild after changing this. |
| `NEXT_PUBLIC_SITE_URL` | Web | `https://beastbots.com` | Canonical site URL used for SEO metadata, sitemaps, and robots.txt. Defaults to `https://beastbots.com`. |

### Infrastructure

| Variable | Where | Default | Description |
|---|---|---|---|
| `NODE_ENV` | Both | `production` | Set automatically by the blueprint. |
| `PORT` | API | `4000` | Port the API listens on. Render routes external traffic to this port. |
| `PORT` | Web | `3000` | Port the Next.js server listens on. |
| `DATABASE_PATH` | API | `/data/beastbot.db` | Path to the SQLite database file. Must point to the persistent disk mount (`/data`). |

## Render-Specific Settings

### Persistent Disk

The `render.yaml` blueprint creates a **1 GB persistent disk** mounted at `/data`. The SQLite database (`beastbot.db`) is stored here so data survives redeploys. Increase the size in the Render dashboard if your dataset grows.

### Health Checks

| Service | Path | Purpose |
|---|---|---|
| API | `/api/health` | Returns DB connectivity status, configured integrations, and memory usage. |
| Web | `/` | Standard Next.js page load check. |

### Build & Start Commands

Both services use the multi-stage Dockerfile, so **no custom build/start commands** are needed in Render—just set the Docker target (already configured in the blueprint).

### Custom Domains

1. Add your domain in the Render dashboard for each service.
2. Update `FRONTEND_URL` on the API to match the Web custom domain.
3. Update `NEXT_PUBLIC_API_URL` on the Web service to match the API custom domain, then **trigger a rebuild** (this is a build-time variable).
4. Optionally set `NEXT_PUBLIC_SITE_URL` on the Web service for SEO metadata.

## Minimal Deployment Checklist

1. **Create both services** via the Render Blueprint.
2. Set `FRONTEND_URL` on the API → Web service URL.
3. Set `NEXT_PUBLIC_API_URL` on the Web → API service URL, then **rebuild** the Web service.
4. Verify `JWT_SECRET` and `ENCRYPTION_KEY` are set (auto-generated by the blueprint).
5. Confirm the API health check passes: `curl https://<api-url>/api/health`.
6. (Optional) Add at least one LLM key (`OAI_API_KEY`, `ANTHROPIC_API_KEY`, or `GROK_API_KEY`) for AI-powered bot decisions.
7. (Optional) Add `RESEND_API_KEY` for email notifications.
8. (Optional) Add platform credentials and webhook secrets for the integrations you plan to use.
